# Design: 初始终端与上位机系统

## 系统架构

系统采用“下位机监测终端 + PC 上位机”的两级结构。下位机基于 ESP32-S3-WROOM-1-N16R8，使用 3 路 NTC 分压采样，其中 2 路用于插排输入线外皮测温，1 路用于环境温度测量。下位机根据采集结果计算导线平均温度、最高温度、稳态或瞬态温升、测点温差和升温速率，并基于这些热特征输出估计电流、过载概率和运行状态。

PC 上位机通过 Wi-Fi 接收下位机发送的 JSON 数据，实时显示温度、电流估计值、过载概率、报警状态和电池电压，并支持参数配置和 CSV 数据保存。终端即使未连接上位机，也应能独立完成采样、判断和本地报警。

## 固件模块

- `ntc_sampler`：ADC 采样、NTC 电阻计算、温度换算和采样校准。
- `feature_extractor`：平均温度、最高温度、温升、测点温差、升温速率等热特征计算。
- `current_estimator`：基于热特征的电流反演接口，初期可使用拟合公式或占位实现。
- `overload_detector`：过载状态判断，初期使用阈值法，后续接入轻量 AI 模型。
- `alarm_controller`：LED 和蜂鸣器报警控制。
- `wifi_transport`：Wi-Fi 连接、实时数据上报、参数接收和断线恢复。
- `power_manager`：电池电压检测、采样周期控制和低功耗策略。
- `config_store`：保存 NTC 参数、报警阈值、采样周期和通信参数。

## 上位机模块

- `connection`：监听或连接终端数据通道。
- `parser`：解析和校验 JSON 遥测数据。
- `charts`：显示温度、温升、估计电流和过载概率曲线。
- `config`：配置采样周期、报警阈值、蜂鸣器开关和标定参数。
- `logger`：将实验过程保存为 CSV 文件。
- `simulator`：在固件未完成前生成模拟遥测数据，用于上位机开发和调试。

## 通信协议

上下位机采用 JSON 文本协议。下位机周期性发送遥测数据，至少包含设备 ID、时间戳、3 路 NTC 温度、导线温度、环境温度、温升、升温速率、估计电流、过载概率、状态、电池电压和 Wi-Fi 信号强度。上位机下发配置数据时，应包含消息类型、参数字段和更新时间。

协议字段的详细定义维护在 `docs/communication-protocol.md`，后续固件和上位机实现必须以该文档为准。

## 硬件约束

- 主控使用 ESP32-S3-WROOM-1-N16R8。
- 测温使用 3 路 NTC 分压采样，优先使用 ESP32-S3 自带 ADC。
- USB-C 用于供电、充电、下载和调试，不作为主要数据链路。
- 主数据链路使用 Wi-Fi。
- 终端使用单节锂电池供电，并设置电源开关。
- 终端包含 LED 和蜂鸣器报警。
- 不使用 OLED 和 SD 卡。
- 夹具只固定在插排输入线外皮上，不进入插排内部。

## 风险与处理

- ESP32-S3 ADC 精度有限，需要通过标定、滤波和参考温度修正降低误差。
- NTC 与线缆外皮接触不稳定会影响热特征，需要在夹具结构上保证压紧和重复定位。
- Wi-Fi 断连不能影响本地报警，因此过载判断和报警必须在下位机独立完成。
- AI 模型第一阶段不宜过重，应先跑通阈值法和接口，再逐步替换为轻量模型。
- 电池续航依赖采样周期、Wi-Fi 占空比和睡眠策略，需要在固件中保留低功耗控制入口。
